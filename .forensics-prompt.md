Every finding gets a stable ID (`F-001`, `F-002`, ...) referenced everywhere. Every artifact is reproducible from a committed script.

## 🤖 AGENT DISPATCH PLAN

Dispatch agents **sequentially** for phases 1–2 (foundation must be solid), then **in parallel** for phase 3 (each finding is independent), then sequentially again for 4–6.

### PHASE 1 — Surface Mapping Agent (1 agent, sequential)

**Mission:** Build a complete, machine-checkable map of the admin attack surface. No opinions, no findings yet — just a faithful inventory.

**Deliverables:**

1. `01-surface-map/routes.json` — every admin route as:

```json
{
	"method": "POST",
	"path": "/admin/users/:id/role",
	"handler_file": "src/admin/users.ts",
	"handler_line": 142,
	"middleware_chain": ["requireAuth", "requireRole:admin", "csrf"],
	"validators": ["UpdateRoleSchema"],
	"db_writes": ["users", "audit_log"],
	"db_reads": ["users", "roles"],
	"external_calls": [],
	"emits_events": ["user.role.changed"]
}
```

Build this by AST/grep, not by reading docs. If a route exists but isn't in the docs, that's evidence too.

2. `01-surface-map/middleware-chain.md` — for each route, the EXACT order of middleware execution, including any conditional middleware. Note any route that bypasses standard chains.

3. `01-surface-map/auth-graph.md` — every distinct authorization check in the codebase, where it's defined, where it's called, and which routes rely on it. Flag any route with NO auth check explicitly.

4. `01-surface-map/db-touchpoints.md` — every raw query, ORM call, and migration touching admin-relevant tables. Note transactions (or lack of), isolation levels, and lock acquisition.

5. `01-surface-map/dependency-graph.svg` — module dependency graph for admin code (use `madge` or equivalent).

**Stop conditions:** Map is complete OR you discover the codebase structure makes mapping impossible — in which case document exactly why and what's missing.

**Output:** Update `00-INDEX.md` with phase 1 summary. Hand off.

### PHASE 2 — Static Forensics Agents (3 parallel agents)

Three specialized agents run in parallel against the surface map. Each writes findings as `F-NNN-<slug>.md` files.

#### Agent 2A — Authorization & Tenant Isolation Auditor

Hunt for:

- Routes with no auth check
- Auth checks that can be bypassed (wrong order in middleware chain, conditionals that fail open, role checks that compare against user-supplied data)
- IDOR — every route that takes an ID and doesn't verify the actor owns/can-access that ID
- Tenant/org isolation failures — queries that don't scope by tenant
- Privilege escalation paths — any place a lower-priv user can mutate higher-priv state
- Impersonation flows — can they be triggered without consent/audit?
- Token/session handling — refresh logic, revocation, fixation, replay
- CSRF gaps on state-changing routes
- Mass assignment — request bodies passed directly to ORM .update()

For EACH finding, write `F-NNN-<slug>.md`:

````markdown
# F-NNN: <one-line title>

**Category:** AUTHZ | IDOR | TENANT | PRIV-ESC | CSRF | MASS-ASSIGN | SESSION
**Severity (hypothesis):** CRITICAL | HIGH | MEDIUM | LOW
**Status:** STATIC-ONLY (awaiting dynamic confirmation)

## Code evidence

File: `src/admin/users.ts:142-160`

```ts
<exact code excerpt>
```

## Why this is a problem

<mechanical explanation — what an attacker does, what the code does in response>

## Reproducer hypothesis

<exact request/sequence that should trigger it — to be executed in Phase 3>

## Confidence

<what would falsify this finding>
````

#### Agent 2B — Data Integrity & Query Auditor

Hunt for:

- Missing transactions around multi-write operations
- Missing row locks (SELECT ... FOR UPDATE) on read-modify-write patterns
- N+1 queries on admin list endpoints
- Missing indexes on admin filter/sort columns
- Soft-delete inconsistencies (some queries filter, others don't)
- Race conditions (TOCTOU between auth check and action)
- Unbounded queries (no pagination, no LIMIT)
- Bulk operations without batching or progress tracking
- Migrations that aren't idempotent or aren't reversible
- Foreign key violations under concurrent load
- JSON column queries without GIN indexes
- ORDER BY without deterministic tiebreaker (causes pagination bugs)

Same `F-NNN` finding format.

#### Agent 2C — Input Validation, Errors, Logging, Side Effects

Hunt for:

- Server-side validation missing or weaker than client
- Validation that runs AFTER the DB write
- Trusting `req.user` fields that came from request body
- File upload: type sniffing only by extension, no size limits, path traversal
- Export endpoints with no row caps
- Error responses leaking stack traces, SQL, internal paths
- Audit log gaps — admin actions not logged, or logged AFTER mutation succeeds (no log on failure paths)
- Audit logs that can be tampered with by the admin themselves
- Background jobs triggered without dedup/idempotency keys
- Webhooks fired before transaction commits
- Cache invalidation missing on admin mutations (stale reads)
- Email/notification side effects in transactions (will fire even on rollback if poorly placed)
- Secrets in logs, error messages, or response bodies

Same `F-NNN` finding format.

**Phase 2 stop condition:** All three agents have walked the entire surface map. Each route in `routes.json` must be marked `audited: true` by all three agents or have a documented reason for exclusion.

### PHASE 3 — Dynamic Confirmation Agents (parallel, one per finding cluster)

Static findings are HYPOTHESES. Phase 3 agents prove or kill them with reproducible artifacts.

For each `F-NNN`:

1. Spin up a clean test environment (use the project's existing test harness — if there isn't one, document that as `F-XXX: NO TEST HARNESS`).
2. Seed deterministic state. Snapshot DB.
3. Execute the reproducer. Capture:
   - Exact command run (`repro.sh`)
   - Raw HTTP request (`request.http`)
   - Raw response, headers + body (`response.txt`)
   - DB diff (`db-state-before.sql` → `db-state-after.sql`)
   - Server logs during the request (`server.log`)
4. Update the finding file with one of:
   - **CONFIRMED** — artifacts attached, severity locked
   - **FALSIFIED** — what the static analysis missed, finding closed
   - **PARTIAL** — reproduces under condition X but not Y, narrowed scope
   - **UNREPRODUCIBLE** — could not set up conditions, explain exactly why

**Adversarial test matrix — every confirmed finding must also be tested against:**

- Concurrent execution (10+ parallel requests)
- Malformed input (fuzzed payloads, oversized fields, null bytes, unicode)
- Network failure mid-request
- DB connection drop mid-transaction
- Partial auth (valid token, wrong role; expired token; revoked token)
- Cross-tenant attempts

If a finding survives all of these, it's locked. If it survives some but not others, the scope is narrowed in the finding file.

**Prior-fix falsification:** Pull every commit message containing `fix`, `patch`, `hotfix`, `security`, `auth`, `admin` from the last 12 months. For each, identify the original failure mode and re-run it against the current code. Log results in `04-falsified-fixes/PRIOR-FIX-<sha>.md`. Any fix that doesn't hold up gets a fresh `F-NNN`.

### PHASE 4 — Severity Ranking Agent (1 agent)

Build `05-severity-matrix.md`:

| ID  | Title | Category | Confirmed | Exploitability | Blast Radius | Data at Risk | Severity |
| --- | ----- | -------- | --------- | -------------- | ------------ | ------------ | -------- |

Severity is computed, not asserted:

- **CRITICAL** — confirmed + remote + no auth required + writes/leaks data across tenants
- **HIGH** — confirmed + requires low-priv auth + crosses trust boundary
- **MEDIUM** — confirmed + requires admin auth but violates least-privilege or audit
- **LOW** — confirmed but limited blast radius or requires unusual conditions

UNVERIFIED findings are listed separately. They do NOT get a severity until confirmed.

---

### PHASE 5 — Implementation Plan Agent (1 agent)

Write `06-fix-plan.md`. Every fix entry must reference the `F-NNN` it resolves and the artifact that proves the bug exists. Generic refactors that don't kill a specific finding are rejected.

Format per fix:

```markdown
## FIX-NNN: <title> (resolves F-XXX, F-YYY)

### Root cause

<mechanical, derived from evidence — not "the code was unclear">

### Why prior approaches failed (if applicable)

<reference to falsified prior fixes>

### Implementation

<complete, production-ready code — no placeholders, no "you can extend this">

### Migration / rollout

- Schema changes (with up + down migrations)
- Backfill plan (if needed)
- Feature flag strategy (if needed)
- Order of deploy steps

### Regression test (REQUIRED)

<exact test added to 07-regression-suite/ that re-runs the original repro and FAILS on unfixed code, PASSES on fixed code>

### Verification commands

<exact commands the human runs to confirm the fix>
```

The plan is sequenced by dependency, not by severity — fixes that share code paths are batched. Each batch is independently deployable.

---

### PHASE 6 — Regression Lock-In Agent (1 agent)

Convert every confirmed finding's repro into an automated test in `07-regression-suite/`. Tests must:

- Live in the project's existing test infrastructure (or document why a new harness is required)
- Be runnable via a single `pnpm` script added to `package.json`
- Reference the `F-NNN` ID in the test name
- Fail on the unfixed code, pass on the fixed code (verify both states)

## 🛑 ANTI-LIE PROTOCOL

Before any agent reports "complete", it must self-audit:

1. Pick 3 random findings from its own output.
2. Re-open the cited file at the cited line numbers. Confirm the code is what was quoted.
3. Re-run the cited reproducer command. Confirm output matches the captured artifact byte-for-byte.
4. If any check fails, the entire agent's output is marked `UNTRUSTED` and re-run from scratch.

Append the self-audit log to the agent's final message.
